### MEDIUM (2+ votes)

#### M1: `tool_timeouts.rs` declared but never used [3 votes: Kimi, Code-Quality, Code-Simplifier]
- `src/fae_llm/tools/tool_timeouts.rs` declares centralized timeout constants (`applescript_exec_secs`, `fetch_url_secs`, `web_search_secs`, `apple_availability_jit_wait_ms`, `apple_availability_poll_ms`) and is added as `pub mod tool_timeouts` in `mod.rs`.
- None of the tool files actually import or use these constants. Each tool file defines its own local `const`.
- This is incomplete refactoring — the module exists as dead code with no users.
- **Action**: Either use the constants in the tool implementations, or defer the module to a future cleanup commit.

#### M2: JIT permission timeout mismatch — `availability_gate.rs` vs `handler.rs` [2 votes: Kimi, Security]
- `availability_gate.rs`: `JIT_TIMEOUT` reduced from 60s → 1200ms. Tool waits max 1.2s on the oneshot channel.
- `handler.rs`: JIT handler still polls the shared permission store for up to **60 seconds** before calling `req.respond_to.send(granted)`.
- Result: The tool will always timeout after 1.2s and return "Permission request timed out" to the LLM, long before the 60s handler resolves the oneshot. JIT permission grants will never succeed.
- The test `jit_timeout_is_fast_and_returns_failure` verifies the fail-fast behavior but does not test a successful grant flow.
- **Confidence**: HIGH — this is a clear race condition between two coupled timeout values that are now mismatched.
- **Action**: Either align the timeouts (both 1.2s or both 60s), or document why the tool-side intentionally fails fast (e.g., JIT is fire-and-forget and permissions are pre-granted synchronously before the tool runs again).
